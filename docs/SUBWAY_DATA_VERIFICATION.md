# 전국 지하철·열차 노선/인접역 데이터 정합성 검증 (LLM 위임용)

다른 LLM에게 통째로 던지세요. 자기완결형 프롬프트입니다.

---

## ROLE

당신은 **한국 도시철도·일반철도 데이터 정합성 검증을 수행하는 시니어 데이터 엔지니어**입니다.
공공데이터·위키백과·각 운영기관 공식 자료(2026년 5월 기준 최신)에 능통하며,
실제 노선도와 운영 정보에 비추어 데이터 오류를 잡아냅니다.

당신의 목표: **현재 데이터셋의 모든 오류·누락·중복을 잡고, 수정된 완전한 데이터셋을 출력**.

---

## SERVICE CONTEXT

서비스: **에어컨 민주주의** (https://aircondemocracy.com)
한국 공공장소(지하철·열차·카페·강의실 등)에서 에어컨 의견을 익명 투표로 모으는 PWA.
사용자가 "방금 지나간 역"과 "다음 도착 예정 역"을 입력하면 노선·방향을 자동 추론.

**핵심 의존성**: adjacency 데이터가 부정확하면
- 실제 인접한 역인데 "인접해 있지 않아요" 오답
- 다른 노선인데 같은 segment로 잘못 묶임
- 사용자 이탈 → 데이터 신호 손실

## 현재 데이터 (검증 대상)

### A. Stations (`src/data/subway-stations.json`)
형식:
```json
[
  { "name": "강남역", "city": "서울", "areas": ["강남구"], "lines": ["2호선", "신분당선"], "lat": 37.4979, "lng": 127.0276 },
  ...
]
```

**현재 커버리지**:
| 도시 | 역 수 | 노선 |
|---|---|---|
| 서울 | 618 | 1~9호선, 공항철도, 수인분당, 경의중앙, 경춘, 신분당, 신림, 우이신설, 의정부선, 김포골드, 인천1/2, 에버라인, 경강, GTX-A, 서해 |
| 부산 | 126 | 1~4호선, 동해선, 부산김해경전철선 |
| 대구 | 88 | 1~3호선 |
| 대전 | 22 | 1호선 |
| 광주 | 20 | 1호선 |
| **총** | **855** (중복 병합 후) | 다수 |

**알려진 문제**:
- 같은 "1호선" 이름이 5개 도시(서울/부산/대구/대전/광주)에 모두 존재 → city 없이는 구별 불가
- "경의중앙" vs "경의중앙선" 표기 혼재 (서울에서 2가지 다 존재)
- "김포 골드라인" vs "김포골드라인" 띄어쓰기 차이
- "신림선" vs "신림역" (오타 의심)
- GTX-A 3 stations만 → 누락
- KTX·SRT·무궁화호·ITX 등 일반철도 데이터 부재

### B. Adjacency (`src/data/subway-adjacency.json`)
형식:
```json
[
  { "from": "강남", "to": "역삼", "line": "2호선" },
  { "from": "강남", "to": "양재", "line": "신분당선" }
]
```

**현재 커버리지**: 723 edges, 22 노선 — **수도권만**.
스테이션 이름은 "역" 접미사 없음 (강남, 봉천 등). 런타임에 정규화 처리.

**커버리지 갭**:
- 부산도시철도 1~4호선 (0 edges)
- 대구도시철도 1~3호선 (0 edges)
- 대전도시철도 1호선 (0 edges)
- 광주도시철도 1호선 (0 edges)
- 부산 동해선·김해경전철선 (0 edges)
- 의정부선·신림선·GTX-A·김포골드라인 일부 (0 edges)
- KTX/SRT/무궁화호/ITX 정차역 그래프 (0 edges)
- city 필드 자체가 없어서 다중-도시 "1호선" 충돌 가능성

## 작업

### Task 1: STATIONS 정합성 검증

`subway-stations.json` 전체를 다음 기준으로 검증:

1. **노선 표기 통일** — "경의중앙" → "경의중앙선", "김포 골드라인" → "김포골드라인" 등
2. **역 이름 정확성** — 위키백과/공식 노선도와 일치 (오타·괄호 표기 차이 잡기)
3. **좌표 sanity check** — 한국 영토 안 (lat 33~38, lng 124~131). 명백히 잘못된 값 표시
4. **누락된 역 식별** — 노선별 공식 역 수와 비교
5. **중복 entry 검출** — 같은 이름 + 같은 좌표(±500m) 다중 entry는 노선 union으로 병합 권장
6. **이름 충돌** — 같은 이름 다른 위치 (양평역 서울 vs 경기) 명시
7. **areas 필드 정확성** — 구·동 단위 (예: 서울대입구역 → 관악구)

### Task 2: ADJACENCY 완전 재생성

`subway-adjacency.json` 을 **0부터 다시 작성**. 다음 기준 충족:

#### 2-1. 형식 변경 (city 필드 필수 추가)
```json
[
  { "from": "강남", "to": "역삼", "line": "2호선", "city": "서울" },
  { "from": "노포", "to": "범어사", "line": "1호선", "city": "부산" },
  { "from": "교대", "to": "동대구", "line": "1호선", "city": "대구" }
]
```

#### 2-2. 커버리지 — 전국 전부
- **수도권**: 서울 메트로 1~9호선, 우이신설선, 신림선, 의정부경전철, 김포골드라인, GTX-A
- **광역**: 공항철도, 수인분당선, 신분당선, 경의중앙선, 경춘선, 경강선, 서해선, 에버라인
- **인천**: 인천1호선, 인천2호선
- **부산**: 1~4호선, 동해선, 부산김해경전철선
- **대구**: 1~3호선
- **대전**: 1호선
- **광주**: 1호선

#### 2-3. KTX/SRT/일반철도 (별도 파일)
일반철도는 별도 `src/data/train-adjacency.json` 으로:
```json
[
  { "from": "서울", "to": "광명", "line": "KTX", "operator": "KORAIL" },
  { "from": "동탄", "to": "지제", "line": "SRT", "operator": "SR" }
]
```

대상 노선:
- **KTX** — 경부선, 호남선, 강릉선, 동해선, 중앙선, 전라선 등 모든 KTX 정차역 chain
- **SRT** — 수서~동탄~지제~천안아산~오송 → 부산/광주
- **ITX-새마을 / ITX-마음** — 주요 정차역
- **무궁화호** — 주요 노선의 정차역 chain (필수: 경부선, 호남선, 중앙선)
- **누리로** — 운행 노선

#### 2-4. 분기 처리
1호선 같은 분기 노선 처리:
- 서울 1호선 = 광운대 ↔ 청량리 / 인천 / 신창 / 광명 등 분기 → 모든 인접 pair 표기
- 5호선 = 방화 ↔ 마천 / 하남검단산 분기 → 모든 pair
- 분기점 (예: 종로3가→을지로4가 in 5호선, 신도림→대림 in 2호선) 검토

#### 2-5. 방향성 / 순환선
- 2호선 외선/내선 순환 → undirected edges로 양방향 모두 매칭 가능해야 함
- 양방향 표시: edge는 1개만 두고, 코드에서 양방향 매칭 (현재 구현 그러함)

### Task 3: 충돌 감지
같은 line 이름이 여러 도시에 있으면 city 필드로 구분.
같은 station 이름이 여러 도시에 있는 경우 별도 검토:
- 광주역 (광주광역시 1호선) vs 광주역 (경강선 광주시) — 도시 다름
- 양평역 (5호선, 서울 양평동) vs 양평역 (경의중앙선, 경기 양평)

## 출력 형식

다음 4개 섹션을 마크다운 문서로:

### 1. Stations 검증 결과

| Issue | 영향받는 entry | 수정안 |
|---|---|---|
| "경의중앙" → "경의중앙선" | 6 stations | lines 필드의 "경의중앙" 모두 "경의중앙선"으로 교체 |
| "신림역" → "신림선" | 1 stations | lines 필드 교체 |
| 봉천역 좌표 오차 | 2 entries | 좌표 통일 |
| ... | | |

### 2. Adjacency 새 dataset (완전 재작성)

`subway-adjacency.json` 형식으로 전체 출력. 다음과 같이 시작:

```json
[
  { "from": "신설동", "to": "동대문운동장", "line": "1호선", "city": "서울" },
  { "from": "동대문운동장", "to": "동대문", "line": "1호선", "city": "서울" },
  ...
]
```

**최소 quality**:
- 모든 수도권 라인 100% (>800 edges 예상)
- 부산/대구/대전/광주 전 노선 100%
- city 필드 모든 row 포함
- 각 edge는 위키백과 노선도 한 번씩 확인

### 3. Train (KTX/SRT/무궁화호) 새 dataset

`train-adjacency.json` 형식 전체 출력.

### 4. 발견된 충돌·이슈 리포트

| 항목 | 설명 | 해결안 |
|---|---|---|
| line "1호선" multi-city | 서울/부산/대구/대전/광주 모두 존재 | city 필드로 구분 |
| ... | | |

## 코드 통합 가이드 (수정 후 적용 절차)

수정된 데이터셋 받으면 다음 적용:
1. `src/data/subway-stations.json` 교체 (필요 시)
2. `src/data/subway-adjacency.json` 교체 (city 필드 포함)
3. `src/data/train-adjacency.json` 새로 생성
4. `src/lib/subwayGraph.ts` 의 `AdjEdge` 인터페이스에 `city: string` 추가
5. `findSegments` 함수에 city 옵션 추가 (현재 station 선택 후 도시 추론)
6. `npm run build` 통과 확인

## 절대 조건

- **추측 금지**. 모르는 노선은 "확인 필요"로 표시. 가짜 데이터 만들지 말 것.
- **모든 edge는 실제 인접** (한 정거장 차이). 인접하지 않은 역 페어는 절대 추가하지 말 것.
- 운영 중단된 노선/역은 **제외**. 신설 예정만 있는 역도 제외 (현재 운행 중인 것만).
- 좌표는 가능하면 정확. 정확도 낮으면 그대로 두기 (좌표 추정 금지).
- 일부 노선(예: 광주2호선)이 건설 중이면 명시 + 데이터 미포함.
