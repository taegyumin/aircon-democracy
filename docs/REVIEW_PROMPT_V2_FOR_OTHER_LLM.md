## 작업 컨텍스트

당신은 **에어컨 민주주의(Aircon Democracy)** 한국 시민 서비스의 강의실 데이터셋(108교 / 114캠 / 624건물)을 **검증·보강**합니다.

**v1 리뷰가 이미 한 번 실패**했습니다. v1 리뷰어는:
- 표본 15교 중 5교만 보강하고 10교는 "추후 보강 필요"라는 escape hatch로 미룸
- 누락 학교 추가 후보 6개 명시했는데 0개 추가
- 보강한 5교 모두 단일 출처에서 일괄 (cross-verification 부재)
- 의심스러운 entry 생성 (POSTECH "LM78", "S02 C5" 등 추측 가능)
- 권위 약한 출처 사용 (중앙대 보안대학원 입학원서 PDF, 경북대 영재교육원 공지)
- 부산대 도메인 `pusan.ac.kr` → `pnu.kr` 무근거 교체

**그래서 v1의 데이터 변경은 전부 롤백**됐고, wizard 버그 fix만 살았습니다. 당신은 v1보다 더 잘 해야 합니다.

## 5대 절대 원칙 (위반 시 commit reject)

### 원칙 1: "추후 보강 필요"는 작업 미완료
- v1 리뷰어가 "공식 캠퍼스맵에 더 많은 건물 확인" 적고 안 고친 학교는 **이번에 반드시 보강**
- 일정 시간 내 확인 안 되면 그 entry는 **건드리지 마세요** (기존 데이터 유지)
- "다음 배치에서" / "추후" / "TBD" / "later" 단어가 들어간 보강 약속은 무효

### 원칙 2: 모든 새 건물 entry는 **2개 이상의 독립 출처에서 일치**해야 함
- 동일 학교 sources URL에 출처 2개 이상 있어야 함
- 같은 도메인(예: pusan.ac.kr의 다른 페이지 두 개)은 1개로 카운트
- 한 출처에서만 나온 건물은 commit 금지 (의심으로 표시 + 별도 PR 코멘트)

### 원칙 3: 권위 우선순위 엄수
공식 캠퍼스맵 > 위키백과 한국어 > 나무위키 > 학교 부속 페이지 > 학생 블로그
- **부속 학원·입학원서·공지사항** 같은 우회 출처는 cross-check 용으로만 사용
- 권위 1순위 출처가 JS-only / image-only면 → **2순위 + 3순위로 cross-verify** (단일 출처 금지)

### 원칙 4: 의심스러운 entry는 즉시 검증
- 코드 패턴 어색하면(예: POSTECH "LM78"의 "LM" zone) → 공식맵에 그 zone 정말 있는지 확인
- 이름이 code와 같으면(예: "S02 C5"의 "C5") → 다른 출처에서 정식 이름 확인
- 확인 안 되면 entry 삭제

### 원칙 5: 도메인·URL 교체 시 commit message에 명시적 이유
- `pusan.ac.kr` → `pnu.kr` 같은 변경은 1줄 이유 필수 ("리다이렉트 확인", "공식 deprecated 안내" 등)
- 이유 없이 교체된 URL은 revert

### 산출물 2: 표본 15교 중 v1 미보강 8교 보강 (필수)
v1이 "추후 보강 필요" 표시한 학교 8개:

### 산출물 3: 의심 entry 검증 결과 (필수)
v1이 생성했다 롤백된 entry 중 의심 항목. 진위 검증 + 결과:

- **POSTECH "LM78"**: LM zone 실존? (실존 시 표준 코드인지 확인)
- **POSTECH "S02 C5"**: code="S02" name="C5"가 정상? 정식 이름?
- **KAIST "E16 정문술빌딩"**: 정문술빌딩 실제 코드 E16 맞는지? N1 아닌지?
- **부산대 도메인 pnu.kr**: pusan.ac.kr이 deprecated 됐는지?
- **중앙대 동번호 101~210**: 보안대학원 PDF 외 공식 출처에서 확인 가능?

각 항목 결과를 `docs/SUSPICIOUS_ENTRIES_VERIFIED.md`에 기록.

### 산출물 4: v1이 못한 cross-source 보강
다음 5개 학교를 **2개 이상 독립 출처**로 cross-check 후 보강. v1의 단일 출처 패턴 안 됨:

- 부산대 (v1: 6→23 단일 출처) — 공식 + 학과 페이지 cross-check 후 재보강
- KAIST 대전 (v1: 4→17 단일 출처) — 공식 PDF + KAIST 위키 cross-check
- POSTECH (v1: 3→22 단일 출처) — 공식 + 부설 페이지 cross-check
- 경북대 (v1: 5→10 단일 출처) — 공식 + 위키 cross-check
- 중앙대 서울 (v1: 12→26 단일 출처) — 공식 + 위키 cross-check

cross-check 결과 양쪽 출처에 일치하는 건물만 채택. v1보다 entry 수가 적게 나와도 OK.

### 산출물 5: 기존 데이터 정확성 표본 검사 (선택, 시간 남으면)
무작위 5교에서 commit된 건물 entry 정확성 확인. 오타·잘못된 이름·존재하지 않는 건물 발견 시 수정. 결과를 `docs/EXISTING_DATA_AUDIT.md`에 기록.

## 절대 금지

- ❌ "추후 보강 필요" / "다음 배치에서" / "TBD" 같은 미루기 표현
- ❌ 한 출처에서만 가져온 건물 추가 (cross-verification 필수)
- ❌ 출처 URL 교체 시 이유 없이 (commit msg에 1줄 이유)
- ❌ 의심 entry 그냥 추가 (검증 결과 commit message에 명시)
- ❌ `main` 체크아웃 / `git push` / `npm install` 추가 패키지
- ❌ 기존 commit 수정·rebase (forward-only commits)
- ❌ 학교 단위가 아닌 bulk commit (학교마다 1 commit)
- ❌ 산출물 1·2·3을 안 하고 산출물 4·5만 함 (우선순위 역전)

## 작업 순서 (이대로 진행)

### Phase 1 (산출물 1+3): 시작 후 ~2시간
1. 누락 학교 6개 후보 각각 web search (학교 공식 사이트 + 위키)
2. 의심 entry 5개 각각 검증 search
3. 결과 docs 작성 + commit (학교당 1 commit + 의심 검증 1 commit)

### Phase 2 (산출물 2): ~3시간
4. 표본 8교 각각 보강 작업 (학교당 1 commit)
   - 2개 출처에서 일치하는 것만 채택
   - 5개 이상 못 찾으면 "검증 불가" 별도 commit

### Phase 3 (산출물 4): ~2시간
5. v1이 단일 출처로 보강했던 5교 cross-check + 재보강

### Phase 4 (산출물 5, 선택): 시간 남으면
6. 기존 데이터 표본 audit

## Commit 정책

- 학교 단위 1 commit: `data(<id>): boost <한글이름> via <출처1> + <출처2>`
- 의심 검증: `verify: <entry명> — <결과>`
- docs 생성: `docs(<주제>): <내용>`
- typecheck 매 commit 후: `cd packages/core && npx tsc --noEmit`

## 자기검증 체크리스트 (commit 전 매번)

이 항목 모두 ✅ 안 되면 commit 금지:

- [ ] 추가한 모든 building에 sources URL ≥2개 있는가?
- [ ] 출처가 모두 다른 도메인인가?
- [ ] commit message에 어떤 학교·어떤 출처 사용했는지 명시했나?
- [ ] `npx tsc --noEmit` 양쪽 (core + web) 통과하는가?
- [ ] "추후" / "TBD" / "later" 단어가 코드/data/commit message에 들어가지 않았나?
