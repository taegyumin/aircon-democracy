# 에어컨 민주주의 — "내 위치 정확히 찝기" + 홈 화면 콘텐츠 UX 자문/설계 (LLM 위임용)

다른 LLM(GPT-5 / Claude Opus / Gemini 2.5 등)에게 통째로 던지세요. 자기완결형 프롬프트입니다.

**두 가지 연계된 UX 문제를 한 번에 다룹니다:**
1. **위치 찝기 여정 단순화** (Part A — 본문)
2. **홈 화면 익명 사용자 콘텐츠 설계** (Part B — 마지막 추가 섹션)

두 문제는 결국 같은 미션 — *5초 안에 정확한 위치로 진입해서 투표 시작* — 의 다른 면.

## ROLE

당신은 **저-마찰 모바일 PWA의 시니어 UX 디자이너 + 인터랙션 디자이너**입니다.
한국 모바일 환경, 대중교통/카페/사무실 등 실내·이동 중 사용 시나리오에 익숙합니다.
당신의 미션: **사용자가 "지금 자신이 있는 정확한 장소"를 찝어내는 여정을 가장 적은 마찰로 설계.**

## SERVICE CONTEXT

### 무엇을 하는 서비스인가
"**에어컨 민주주의**" (https://aircondemocracy.com) — 한국 공공장소 에어컨 분위기를 익명 투표로 모으는 PWA.
선택지 3개: `추워요` / `적당해요` / `더워요`. 한 장소·사용자당 1표, 30초 쿨다운, 1시간 자동 만료.

### **핵심 제약: 5초 안에 투표 가능해야 함**
이게 본 서비스의 진짜 가치.
- 사람이 추워 죽겠어서 앱을 켜는 순간 → 가능한 한 빨리 표 던지고 다시 핸드폰 넣게.
- 사용자가 5초 넘게 헤매면 그냥 안 함. 데이터 없으면 서비스 무의미.

### 사용 시나리오 (피크 사용 컨텍스트)
1. **출퇴근 지하철 2호선 강남→삼성** — 사람 미어터지고, 한 손, 핸드폰 들고 흔들림
2. **점심시간 스타벅스 강남대로점** — 좌석에 앉아 있음. 비교적 안정
3. **강의 시작 직전 강의실** — 친구와 잡담 중, 빨리 처리
4. **퇴근길 KTX 부산행 3호차** — 좌석. 시간 있음. 정밀 입력 가능
5. **272번 버스 신촌오거리 정류장** — 서서 기다림. 한 손

## CURRENT FLOW (현재 UI — 비판/개선 대상)

### Step 0: 홈
- 큰 파란 CTA: `[📍] 지금 어디 계세요? · 지하철 · 버스 · 강의실 · 사무실 · 기타 [>]`
- 그 외: 검색바 (작음), QR 버튼 (비활성), 장소 카드 리스트 (의견 모이는 중 / 장소)

### Step 1 (CTA 누르면): 카테고리 선택
5개 카드 세로 리스트, 각 카드에 Lucide 아이콘 + 색상 톤:
- 🚇 지하철 (도시철도) — 파랑
- 🚄 기차 (KTX·SRT·무궁화호 등) — 빨강
- 🚌 버스 (시내·시외) — 초록
- 🎓 강의실 (학교) — 보라
- 🏢 사무실 (회사) — 슬레이트
- 📍 기타 (카페·도서관 등) — 오렌지

### Step 2A: 지하철 선택 시
- "어느 노선 타고 계세요? *" — 29개 노선 색상 칩 (1호선~9호선, 신분당, 공항철도, GTX-A, 부산김해경전철 등 다 노출)
- "몇 번 칸이에요? *" — 1~10 그리드 + "잘 모르겠어요" 버튼
- "지나고 있는 역 (선택)" — 노선 기반 자동완성 (한글 초성 검색 OK)
- "투표하러 가기" 버튼

### Step 2B: 기차 선택 시
- "어떤 열차 타고 계세요? *" — 6개 (KTX, SRT, ITX-새마을, ITX-마음, 무궁화호, 누리로)
- "몇 호차예요? *" — 1~20 그리드 + 모름
- "어디까지 가세요? (선택)" — 텍스트
- "투표하러 가기"

### Step 2C: 버스 선택 시
- "노선 번호?" — 텍스트 입력 (예: 272, 5511, M7106)
- "정류장? (선택)" — 텍스트
- "투표하러 가기"

### Step 2D: 강의실/사무실/기타
- 별도 등록 화면(RegisterScreen)으로 forward, 자유 텍스트 입력

### Step 3: 투표 화면
- 큰 3개 버튼 (추워요 / 적당해요 / 더워요)
- 누르면 즉시 반영

### 현재 마찰 측정 (지하철 시나리오)
홈 도착 → 투표 완료까지:
1. CTA 탭 (1)
2. "지하철" 선택 (2)
3. "2호선" 칩 탭 (3) — 29개 중 찾기
4. "8번 칸" 그리드 탭 (4) — 칸 모를 가능성 높음
5. (선택) 역 검색 입력 + 탭 (5~6)
6. "투표하러 가기" (7)
7. "추워요" 탭 (8)

**최선 8탭, 최악 10탭 + 텍스트 입력.** 핸드폰 흔들리는 지하철에서 그러기엔 너무 많음.

## 사용 가능한 신호 (UX가 활용할 수 있는 것들)

### 1. 브라우저 / OS 기본
- **Geolocation API** — GPS + WiFi triangulation. 지하 깊은 곳/지하철 안에선 잘 안 잡힘. 권한 prompt 필요.
- **localStorage / 쿠키** — 최근 본 장소, 최근 카테고리
- **navigator.connection** — 네트워크 종류 (4G/5G/WiFi)
- **DeviceOrientation, Acceleration** — 흔들림 패턴으로 "차량 이동 중" 추정 가능
- **navigator.languages** — 한국어 사용자 판단
- **Service Worker 캐시** — 오프라인 fallback

### 2. 서버측 신호
- **요청 IP** — 캐리어 NAT, 회사/학교 WiFi IP는 추정에 도움
- **요청 시간** — 출퇴근 시간 / 점심시간 / 강의시간 패턴
- **이전 voter_id 활동 이력** — 보통 같은 시간대 같은 장소 가능성

### 3. 외부 데이터 (이미 보유)
- **전국 지하철역 874개 DB** — name, city, areas, lines, lat/lng (수도권·5대 광역시)
- **장소 풀 (모든 사용자 등록 장소)** — 인기 카페/대학/사무실 누적
- **시간별 인기 장소** — 향후 가능

### 4. 외부 데이터 (잠재)
- **카카오맵/네이버맵 Place Search API** — 유료. 일일 무료 한도 제한적
- **KORAIL 실시간 열차 정보** — 공공데이터포털, 무료, key 필요
- **서울교통공사 실시간 지하철 위치** — 공공데이터포털, 무료
- **공공와이파이 SSID DB** — 일부 공공장소 식별 가능

### 5. 사용자 행동
- **NFC 태그 스캔** — 상점/지하철 객실에 스티커 (인프라 비용)
- **QR 코드 스캔** — 상점 등록 시 자동 생성 가능 (UI는 이미 있음, 비활성 상태)
- **수동 즐겨찾기** — 자주 가는 장소 단축

## 사용자 페르소나

### A. 김출근 (32세, 회사원)
- 매일 같은 출근길: 사당 → 강남 (2호선)
- 매일 점심: 회사 근처 스타벅스
- 매일 퇴근: 같은 코스
- → **반복 시나리오. "어제와 같음" 추정이 강력.**

### B. 박학생 (21세, 대학생)
- 강의실 일주일에 5번, 매번 다른 교실
- 카페·도서관 이동 빈번
- 친구와 만나는 곳 매번 다름
- → **변동 시나리오. 사전 정보 적음. 검색이 중심.**

### C. 이여행 (40세, 출장)
- KTX 자주 탐, 매번 다른 호차/방향
- 회사 출장 = 처음 가는 빌딩/카페
- → **신규 컨텍스트. GPS + 검색 의존.**

### D. 정주민 (65세, 시민)
- 동네 도서관/노인정/지하철
- 스마트폰 다루기 서툴음, 큰 글씨 선호
- → **저빈도 사용, 접근성 중요.**

- iOS Safari + Android Chrome 둘 다 OK
- 다크 모드 대응 (현재 라이트만)

- 한국어 한글 가독성 (특히 작은 폰트 회피)

### 비선호

- 텍스트 입력 강제 (지하철 흔들림 + 한 손)

## 출력 요청

다음 8개 섹션을 마크다운으로 작성하세요. 각 섹션은 즉시 실행 가능한 수준의 디테일.

### 1. 현재 UX 진단 (Critique)
- 5초 안에 투표 가능한가? 시나리오 4개(위 페르소나)별 평가
- 마찰 포인트 top 5 (어디서 사용자가 멈출 가능성 높은가)
- 인지 부하 분석 (29개 노선 칩 한 번에 보여주기 = 적정한가?)
- 모바일 제스처 + 한 손 도달 영역 평가

### 2. 권장 핵심 UX 변화 (Top 3)
가장 임팩트 큰 3가지 변경. 각각에 대해:
- 무엇을
- 왜 (이론적 근거 + 데이터/사례)
- 어떻게 (ASCII 모크업 또는 자세한 step-by-step 묘사)
- 트레이드오프

### 3. 컨텍스트 추론 전략
- Geolocation 권한 언제 어떻게 요청? (소프트 ask 패턴 권장)
- 시간 + 위치 + 이력 → "추정 장소" 도출 알고리즘
- 추정 결과 보여주는 UI 패턴 (확인 vs 자동 선택)
- 추정 실패 시 fallback
- 지하 깊은 곳 / 빠르게 이동 중 / 새 위치 처리

### 4. 페르소나별 권장 흐름 (Flow Spec)
페르소나 A~D 각각:
- 첫 진입 vs 두 번째 진입 vs N번째 진입
- 평균 탭 수 / 시간 (목표 vs 현재)
- ASCII 시퀀스 다이어그램

### 5. 컴포넌트 변경 명세
- HomeScreen: 어떻게 바뀌어야 하는가
- LocationWizardScreen: 어떻게 바뀌어야 하는가
- 새로 필요한 컴포넌트가 있다면 (예: QuickResumeCard, NearbyPlacesSheet)
- 각 컴포넌트의 입출력, 상태, 디스플레이 규칙

### 6. 카테고리별 마이크로-UX
지하철/기차/버스/강의실/사무실/카페 각각:
- 가장 짧은 path
- 흔한 mistake와 그것을 회복하는 UI
- "잘 모르겠음" 케이스 처리

### 7. 마찰 측정 메트릭 (KPI)
- 어떤 지표를 추적해야 UX 개선 효과 검증 가능?
- 어떻게 측정?  (Cloudflare Analytics? 자체 이벤트?)
- 정상 baseline 추정

### 8. 점진적 도입 로드맵
- 이번 주 (P0): 코드 변경 최소, 큰 마찰 제거 1~2개
- 이번 달 (P1): Geolocation 통합, 추정 UI, 즐겨찾기
- 분기 단위 (P2): NFC/QR 인프라, 외부 API 통합, 머신러닝 추정

# Part B: 홈 화면 익명 사용자 콘텐츠 설계

## 추가 문제 정의

홈 화면은 사용자가 가장 먼저 보는 곳. 현재 구조:
1. 상단 검색바 + (비활성) QR 버튼
2. 큰 파란 CTA: "지금 어디 계세요?" → 위치 위저드
3. "장소" 섹션 — 모든 장소 리스트 (현재 시드 6개, 전부 "의견 없음")
4. 하단: "장소가 없나요? 직접 등록하기" 배너

### 핵심 질문
**로그인 안 한 사용자(=익명)에게 홈에 뭘 보여주는 게 가장 좋을까?**

조건:
- 사용자 입장에서 유익해야 함
- 더 많은 참여를 유도해야 함
- 앱의 본질(빠른 익명 투표)을 해치지 않아야 함

### 운영자의 인사이트 (참고용)
1. **익명도 voter_id 쿠키 1년 발급됨** — 즉 "최근 본 장소 단말 기억"은 로그인 없이도 가능
2. 로그인 = "기기 간 동기화 + 영구 즐겨찾기" 정도로 차별화
3. 현재의 "모든 장소 의견 없음" 빈 상태는 죽은 앱처럼 보임 — 해결 필요
4. 다른 사람 vote 보고 본인 의견 흔들리는 anchoring 위험 (특히 첫 vote 전)
5. 데이터 정직성 우선. 가짜 활동/트렌딩으로 채우면 안 됨

### 운영자가 고려한 옵션 (LLM이 비판/통합/추가 환영)

| 옵션 | 가치 | 위험 |
|---|---|---|
| A. 지금 활성 장소 (모이는 곳) | FOMO, 사회적 증거 | 빈 상태 시 빈약 |
| B. 익명 voter의 최근 본 장소 | 반복 사용성 | 신규 유저엔 없음 |
| C. 카테고리 단축 (지하철/카페 큰 칩) | 1탭 단축 | CTA와 중복 |
| D. 위치 기반 추천 (geolocation soft ask) | 가장 강력 | 권한 마찰 |
| E. 통계 / 사회적 증거 ("지난 24h 6,432명") | 신뢰감 | 초기엔 의미 X |
| F. 트렌딩 (가장 추웠던 곳 TOP) | 흥미 유발 | "보복 vote" 유도 위험 |
| G. 둘러보기 (강남/홍대 등 인기 지역) | 탐색 욕구 | 본질 흐릿 |
| H. 첫 사용자 가이드 | 학습 | 매번 보면 짜증 |
| I. SEO용 풍부한 콘텐츠 | 검색 노출 | 사용자 가치는 별개 |
| J. 강한 로그인 유도 배너 | DAU 향상 | 익명성 가치 훼손 ❌ |

### 운영자의 잠정 추천 (LLM이 검증/뒤집어도 됨)

```
┌─────────────────────────────────────┐
│  ◎  지금 어디 계세요?    [기존 CTA]  │ ← 항상 최상단
└─────────────────────────────────────┘

[다음 섹션들 — 데이터 있을 때만 노출, 빈 섹션 숨김]

🔥 지금 의견이 모이고 있어요 (최대 5개)
   - 활성 vote 있는 장소만, 참여자 많은 순
   - 빈 상태면 섹션 자체 숨김

⏱ 최근에 봤던 곳 (쿠키 기반, 익명도 OK)
   - voter_id가 30일 내 본 장소
   - 신규 유저면 숨김

🚇 빠른 진입
   [2호선] [9호선] [공항철도] ... ← 인기 노선 칩

📍 (선택) 근처 장소 보기 — soft ask
   "[허용]  [무시]"

🌡 (선택) 오늘의 통계
   "지난 24h 6,432명이 의견을 남겼어요"

+ 장소 없나요? 직접 등록하기
```

기본 원칙: **빈 섹션은 숨김, 데이터 쌓일수록 화면 풍부해짐.**

## Part B 출력 요청

Part A의 8개 섹션에 더해서 다음 4개 섹션 추가:

### 9. 홈 화면 콘텐츠 전략 비평
- 운영자 잠정 추천에 대한 비판 + 동의 (각 섹션별)
- 빈 상태 vs 데이터 풍부 상태 시각화 (모크업)
- 위 옵션 매트릭스(A~J)에서 LLM 본인 추천 조합 + 이유
- 추천 안 한 옵션의 명시적 거절 이유

### 10. 신규 vs 회귀 사용자 분기
- 처음 진입 (voter cookie 없음): 무엇 보여줄지
- N번째 진입 (cookie 있고, 최근 본 장소 있음): 어떻게 다르게
- 시간대별 변화 (출퇴근/점심/저녁)
- 익명 vs 로그인 차이 (운영자 표 검증 또는 보완)

### 11. 콘텐츠 노출 규칙 (Visibility Rules)
각 섹션의 노출 조건을 명확히. 예시:
- "지금 모이는 곳" 노출 조건: 활성 vote ≥ 3명인 장소가 최소 1개 있을 때
- 위치 권한 soft ask: 진입 N회 이후, 카테고리 단축 탭한 적 없을 때
- 통계: 총 vote 수 > 1000 누적 이후
- 등등

### 12. 데이터 부족 초기 (Cold Start) 전략
출시 직후 voter도 적고 활성 vote도 거의 없을 때:
- 어떻게 "죽은 앱" 인상을 피할까
- 시드 장소만 6개 있을 때 무엇 보여줄지
- 첫 활성 사용자 N명에게 어떤 차별화된 메시지 (예: "당신이 1번 의견자!")
- 초기 사용자 행동 유도 (registration nudge 등)

## Part B 미션 한 줄

> "익명 첫 진입자도 첫 화면에서 '이 앱 살아 있고 나한테 유익하다' 느끼게 하라. 단, 본질(빠른 투표·익명성)을 1mm도 훼손하지 말 것."

## 최종 통합 가이드 (Part A + B)

LLM의 최종 산출물은 다음을 모두 포함:
- Part A의 12개 섹션 (8 + 4)
- 두 Part가 충돌하는 부분 (예: 위치 권한 ask 타이밍)은 한 가지로 통합 결정
- 권장 변경의 P0/P1/P2 우선순위는 두 Part 통합해서 단일 로드맵

분량: 4000~7000단어 (Part B 포함 증가).
